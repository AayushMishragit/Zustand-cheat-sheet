01 Core Concepts
What is Zustand?

Zustand (German for "state") is a small, fast, and scalable state management library for React. It uses a simplified flux pattern — a centralized store with actions — but with almost zero boilerplate. State lives outside React's tree, components subscribe to it, and re-render only when relevant slice changes.

Zero Provider No Reducers Unopinionated TypeScript First
Why use Zustand?

React's built-in state (useState, useReducer) causes prop drilling and unnecessary re-renders. Context API re-renders all consumers on any change. Zustand solves this with atomic subscriptions — each component re-renders only when its specific slice changes.

Granular Re-renders No Context Hell Devtools Support
When to use Zustand?

Use Zustand when state is shared across distant components (sibling or cousin), when server state alone (React Query/SWR) isn't enough for UI state, or when you need complex interactions like undo/redo, multi-step forms, or canvas editors.

✓ Shared UI state ✓ Cross-component data ✗ Server cache (use RQ) ✗ Component-local state

02 Store Creation
basic store
import { create } from 'zustand'

// create() takes a function that returns state + actions
const useCounterStore = create((set) => ({
  // --- STATE ---
  count: 0,

  // --- ACTIONS ---
  increment: () =>
    set((state) => ({ count: state.count + 1 })),

  decrement: () =>
    set((state) => ({ count: state.count - 1 })),

  reset: () => set({ count: 0 }),
}))
set() and get() patterns
const useStore = create((set, get) => ({
  items: [],
  filter: 'all',

  // set() merges state (shallow)
  setFilter: (f) => set({ filter: f }),

  // set() with updater fn for derived logic
  addItem: (item) => set((state) => ({
    items: [...state.items, item]
  })),

  // get() reads current state in actions
  getFilteredItems: () => {
    const { items, filter } = get()
    return filter === 'all'
      ? items
      : items.filter(i => i.status === filter)
  },
}))
Pro tip: set() does a shallow merge — you don't need to spread the whole state. Only return the keys you want to update.
Warning: For nested state (objects inside objects), always spread explicitly: set(state => ({ user: { ...state.user, name: 'Alice' } }))
03 Common State Patterns 
const useListStore = create((set) => ({
  items: [], // typed: Item[]

  // ADD
  addItem: (item) =>
    set((state) => ({ items: [...state.items, item] })),

  // REMOVE by id
  removeItem: (id) =>
    set((state) => ({
      items: state.items.filter((i) => i.id !== id)
    })),

  // UPDATE by id
  updateItem: (id, patch) =>
    set((state) => ({
      items: state.items.map((i) =>
        i.id === id ? { ...i, ...patch } : i
      ),
    })),

  // REORDER (drag and drop)
  reorderItems: (from, to) =>
    set((state) => {
      const arr = [...state.items]
      arr.splice(to, 0, arr.splice(from, 1)[0])
      return { items: arr }
    }),
}))
const useUserStore = create((set) => ({
  user: {
    id: null,
    name: '',
    preferences: { theme: 'dark', lang: 'en' },
  },

  // Partial update — spread carefully!
  updateUser: (patch) =>
    set((state) => ({
      user: { ...state.user, ...patch }
    })),

  // Nested object update
  setTheme: (theme) =>
    set((state) => ({
      user: {
        ...state.user,
        preferences: { ...state.user.preferences, theme },
      },
    })),

  clearUser: () => set({ user: { id: null, name: '', preferences: {} } }),
}))
const useSelectionStore = create((set) => ({
  items: [],
  selectedId: null,         // single select
  selectedIds: new Set(), // multi-select

  select: (id) => set({ selectedId: id }),
  deselect: () => set({ selectedId: null }),

  // Toggle multi-select
  toggleSelect: (id) =>
    set((state) => {
      const next = new Set(state.selectedIds)
      next.has(id) ? next.delete(id) : next.add(id)
      return { selectedIds: next }
    }),

  // Derived: get the selected item object
  getSelected: () =>
    get().items.find(i => i.id === get().selectedId),
}))
const useUIStore = create((set) => ({
  isModalOpen: false,
  isSidebarOpen: true,
  activeTab: 'overview',
  toasts: [],
  isLoading: false,

  openModal: () => set({ isModalOpen: true }),
  closeModal: () => set({ isModalOpen: false }),
  toggleSidebar: () =>
    set((s) => ({ isSidebarOpen: !s.isSidebarOpen })),

  addToast: (msg) =>
    set((s) => ({
      toasts: [...s.toasts, { id: Date.now(), msg }]
    })),
}))
04 Action Patterns
CRUD + Async Actions
const useTaskStore = create((set, get) => ({
  tasks: [],
  loading: false,

  // ADD
  addTask: (title) => set((s) => ({
    tasks: [...s.tasks,
      { id: crypto.randomUUID(), title, done: false }]
  })),

  // UPDATE
  updateTask: (id, patch) => set((s) => ({
    tasks: s.tasks.map(t => t.id===id ? {...t,...patch} : t)
  })),

  // DELETE
  deleteTask: (id) => set((s) => ({
    tasks: s.tasks.filter(t => t.id !== id)
  })),

  // TOGGLE
  toggleTask: (id) => set((s) => ({
    tasks: s.tasks.map(t =>
      t.id===id ? {...t, done: !t.done} : t)
  })),

  // ASYNC fetch
  fetchTasks: async () => {
    set({ loading: true })
    const data = await fetchAPI('/tasks')
    set({ tasks: data, loading: false })
  },
}))
Select Item Pattern
const useCanvasStore = create((set, get) => ({
  shapes: [],
  selectedId: null,

  selectShape: (id) => set({ selectedId: id }),

  // Get derived data without storing it
  getSelected: () => {
    const { shapes, selectedId } = get()
    return shapes.find(s => s.id === selectedId)
  },

  moveShape: (id, dx, dy) =>
    set((s) => ({
      shapes: s.shapes.map(sh =>
        sh.id === id
          ? {...sh, x: sh.x+dx, y: sh.y+dy}
          : sh
      )
    })),
}))
get() tip: Use get() inside async functions to read latest state — never close over stale state from the outer function scope.
05 Using Store in React Components
Selectors — read only what you need
// ✅ GOOD — only re-renders when count changes
const count = useCounterStore(
  (state) => state.count
)

// ✅ GOOD — multiple values with stable selector
const { user, updateUser } = useUserStore(
  (state) => ({
    user: state.user,
    updateUser: state.updateUser
  }),
  shallow // shallow equality check
)

// ❌ BAD — re-renders on ANY store change
const store = useCounterStore() // don't do this

// ✅ Actions don't need shallow
const increment = useCounterStore(
  (s) => s.increment // stable reference
)
Performance Optimization
import { useShallow } from 'zustand/react/shallow'

// useShallow — compare arrays/objects by value
function TaskList() {
  const tasks = useTaskStore(
    useShallow((s) => s.tasks)
  )
  // re-renders only when tasks array changes
}

07 Real Project: Figma-like Canvas Store 
import { create } from 'zustand'
import { devtools, persist } from 'zustand/middleware'

type ShapeType = 'rect' | 'circle' | 'text'
type Tool    = 'select' | 'rect' | 'circle' | 'text' | 'pan'

interface Shape {
  id: string; type: ShapeType
  x: number; y: number; w: number; h: number
  fill: string; opacity: number; locked: boolean
}

interface CanvasStore {
  shapes: Shape[]; selectedIds: Set<string>
  activeTool: Tool; zoom: number; pan: { x: number; y: number }
  history: Shape[][]
  addShape: (shape: Omit<Shape, 'id'>) => void
  moveShape: (id: string, dx: number, dy: number) => void
  resizeShape: (id: string, w: number, h: number) => void
  deleteSelected: () => void
  setTool: (tool: Tool) => void
  setZoom: (z: number) => void
  undo: () => void
}

export const useCanvasStore = create<CanvasStore>()(
  devtools(
    persist(
      (set, get) => ({
        shapes: [], selectedIds: new Set(),
        activeTool: 'select', zoom: 1,
        pan: { x: 0, y: 0 }, history: [],

        // snapshot for undo before mutations
        _snapshot: () => set((s) => ({
          history: [...s.history.slice(-20), s.shapes]
        })),

        addShape: (shape) => {
          get()._snapshot()
          set((s) => ({
            shapes: [...s.shapes,
              { ...shape, id: crypto.randomUUID(), locked: false }]
          }))
        },

        moveShape: (id, dx, dy) =>
          set((s) => ({
            shapes: s.shapes.map(sh =>
              sh.id===id && !sh.locked
                ? {...sh, x: sh.x+dx, y: sh.y+dy} : sh)
          })),

        resizeShape: (id, w, h) =>
          set((s) => ({
            shapes: s.shapes.map(sh =>
              sh.id===id ? {...sh, w, h} : sh)
          })),

        deleteSelected: () => {
          get()._snapshot()
          set((s) => ({
            shapes: s.shapes.filter(sh => !s.selectedIds.has(sh.id)),
            selectedIds: new Set()
          }))
        },

        undo: () => set((s) => {
          const prev = s.history[s.history.length - 1]
          if (!prev) return s
          return {
            shapes: prev,
            history: s.history.slice(0, -1)
          }
        }),

        setTool: (activeTool) => set({ activeTool }),
        setZoom: (zoom) => set({ zoom: Math.max(0.1, Math.min(8, zoom)) }),
      }),
      { name: 'canvas-v1', partialize: (s) => ({ shapes: s.shapes }) }
    ),
    { name: 'CanvasStore' }
  )
)
08 Architecture Flow
UI Layer

Toolbar · Panels · Canvas

→
Actions

addShape · move · resize

→
Zustand Store

shapes[] · selectedIds · tool

→
Canvas Render

React / Konva / WebGL

UI → Actions

User clicks toolbar → calls setTool('rect'). Drags on canvas → calls addShape({x,y,w,h}). Clicks shape → calls selectShape(id). Each component calls one clean action — no business logic in the component.

Store → Canvas Rendering

Canvas component subscribes to shapes via selector. When any shape changes, only the canvas re-renders. The toolbar subscribes to activeTool only — it never re-renders on shape changes. Fine-grained subscriptions = 60fps guaranteed.

History / Undo Pattern

Before any destructive action, call _snapshot() to push current shapes to a history stack (capped at 20). undo() pops the last snapshot and sets it as current state. No external library needed — Zustand handles it naturally.

Multi-store Composition

Keep concerns separate: canvasStore for shapes/selection, uiStore for panels/modals, userStore for auth/preferences. Stores can read each other via useOtherStore.getState() in actions.

10 Best Practices
  • 01Keep stores small and focused. One store per domain (users, tasks, canvas, UI). Avoid one giant store — it's harder to maintain and test.
  • 02Always use selectors when consuming store in components. Never subscribe to the whole store — useStore() without a selector re-renders on any change.
  • 03Put actions inside the store, not in components. Actions are just functions in the store creator. Keeps components dumb and logic testable.
  • 04Use get() inside async actions to read latest state. Closing over stale state from the outer function will cause bugs in async flows.
  • 05Use useShallow when selecting multiple values. Without it, a new object literal on every render defeats memoization.
  • 06Name your actions semantically. setIsModalOpen(true) is worse than openModal(). Actions should express intent, not expose raw state setters.
  • 07Don't store derived data. If a value can be computed from existing state, compute it in a selector or a getter method — don't store it redundantly.
  • 08Type your stores with TypeScript interfaces. Define the store shape as an interface and pass it as the generic: create<MyStore>().
  • 09Use devtools in development and name your action calls. The third arg to set() is an action name string visible in Redux DevTools.
  • 10Use partialize with persist to exclude ephemeral state. Don't persist loading flags, error states, or non-serializable values like Sets.
  • 11Test stores independently of React. Zustand stores are plain JS — import and test them with Vitest/Jest without rendering any components.
  • 12Reset store state between tests. Call useStore.setState(initialState) in beforeEach to ensure test isolation.
11 Interview Quick Answers
What is Zustand and how does it differ from Redux? ▾
Zustand is a minimal state management library using a hook-based API. Unlike Redux, it requires no Provider, no reducers, and no action creators. You create a store with create(), define state and actions in one object, and consume it with a selector hook anywhere in the tree. Redux enforces strict patterns good for large teams; Zustand is flexible and fast to set up.
How does Zustand avoid unnecessary re-renders? ▾
Zustand uses reference equality to determine if a component should re-render. When you use a selector — useStore(s => s.count) — the component only re-renders if that specific slice changes. For objects/arrays, use useShallow for shallow comparison. Without a selector, the component subscribes to the entire store and re-renders on every change.
What is the difference between set() and get()? ▾
set() updates the store state — it merges the returned object shallowly. You can pass an object or an updater function set(s => ({...})) for derived updates. get() reads the current state synchronously at the time of the call — essential inside async actions to avoid closing over stale state from the closure.
How do you handle async actions in Zustand? ▾
Async actions are just async functions in the store. Set a loading flag before the await, fetch data, then set the result. Use get() to read fresh state after awaits. No thunks or sagas needed — it's plain JavaScript. Example: fetchUsers: async () => { set({loading:true}); const data = await api.getUsers(); set({users:data,loading:false}) }.
How does the persist middleware work? ▾
Wrap your store creator with persist(creator, options). It automatically saves state to localStorage (or any storage) on each change and rehydrates on load. Use partialize to only persist specific keys (exclude loading states, functions, Sets). Use version + migrate for schema migrations when your state shape changes between releases.
Can Zustand be used outside React components? ▾
Yes — this is one of Zustand's superpowers. Call useMyStore.getState() to read state and useMyStore.setState() to update it anywhere: WebSocket listeners, service workers, utility functions, or even in other frameworks. Use useMyStore.subscribe(selector, callback) to react to changes outside React.
When should you NOT use Zustand? ▾
Don't use Zustand for server state (use React Query or SWR instead — they handle caching, refetching, and invalidation). Don't use it for component-local state that doesn't need to be shared (useState is fine). Don't use it when you need strict, enforced patterns for a large enterprise team — Redux Toolkit's conventions may serve better there.
How do you test a Zustand store? ▾
Zustand stores are plain JavaScript — import the store, call actions, and assert on state. In beforeEach, reset state with useStore.setState(initialState, true) (the second true arg replaces instead of merges). You can test stores entirely without React. For component tests, wrap in a test Provider or mock the module.
